# Codex PR 종합 리뷰 자동 요청 운영

상태: `accepted`

## 배경
- 5개 자동 PR 리뷰 코멘트는 관점은 분명하지만, PR 대화에 같은 작성자가 연속으로 등장해 누가 어떤 기준으로 말하는지 오히려 보기 어렵다.
- 자동 요청 수가 많으면 리뷰 시작점은 명확해도 대화 UX 가 산만해지고, 기본 리뷰 루틴이 과하게 무겁게 보인다.
- 저장소는 이미 `AGENTS.md`, `docs/`, hook, `verify` 로 운영 규칙을 구조화하고 있으므로, 리뷰도 요청 의도를 PR 대화에 직접 남기는 방식이 더 잘 맞는다.
- GitHub 와 연결된 Codex OAuth 댓글 경로를 기본값으로 쓰므로, 저장소 안의 별도 API key 나 `openai/codex-action` workflow 는 유지하지 않는다.
- 다만 `GITHUB_TOKEN` 으로 남긴 댓글은 `github-actions[bot]` 주체가 되어 Codex 응답 경로와 맞지 않는다.
- 그래서 `Verify` 뒤 자동 댓글이 필요하면, Codex 와 연결된 같은 GitHub 계정의 PAT 를 workflow secret 으로 사용해야 한다.

## 결정
- GitHub 와 연결된 Codex OAuth 댓글 경로를 기본 리뷰 경로로 사용한다.
- `Verify` workflow 의 `request_codex_review` job 은 복구용으로 보존하되 현재는 `if: ${{ false }}` 로 비활성화한다.
- 자동 리뷰 요청을 복구할 때는 `Verify` workflow 가 `pull_request` 에서 성공하면 후속 job 이 `COMMENTER_PAT` 로 기본 종합 리뷰 요청 코멘트 1개를 남긴다.
- `COMMENTER_PAT` 는 Codex 와 연결된 같은 GitHub 계정의 토큰을 사용한다.
- 기본 종합 리뷰는 `architecture`, `security`, `docs`, `feature`, `qa` 5개 관점을 한 댓글 안의 고정 섹션으로 요청한다.
- 자동 코멘트는 `head.sha + composite` 기준 marker 를 남겨 같은 커밋에 중복 요청이 쌓이지 않게 한다.
- 자동 코멘트는 `리뷰 범위`, `5개 관점 섹션`, `낮은 우선순위`, `출력 형식` 4개 블록으로 구성해 응답 구조를 고정한다.
- 추가 요청은 PR 코멘트 `@codex <관점과 지침>` 형태로 남기고, 필요하면 `reliability`, `performance`, `testing` 같은 관점을 직접 적어 확장한다.
- 리뷰 원칙은 루트 `AGENTS.md` 의 `Review guidelines` 에 두고, 세부 지시는 기본 종합 코멘트에서 직접 전달한다.

## 대안
- 섹션 구조를 강제하지 않는 단일 자동 리뷰는 관점 혼합이 심해지고, 리뷰 의도가 다시 모호해질 수 있어 채택하지 않았다.
- GitHub label 로 profile 을 고르는 방식은 보기에 직관적이지만, 라벨 상태 관리 비용이 커서 채택하지 않았다.
- 커스텀 GitHub Actions workflow 와 profile prompt 를 유지하는 방식은 운영 복잡도와 인증 경로 혼선을 키워 채택하지 않는다.
- `GITHUB_TOKEN` 으로 `@codex` 댓글을 남기는 방식은 댓글 작성 주체가 `github-actions[bot]` 가 되어 사용자 계정 응답 경로와 어긋나므로 채택하지 않는다.
- 사람이 매번 종합 코멘트를 수동으로 남기는 방식은 단순하지만, 기본 리뷰 루틴이 자주 빠져 자동화를 유지한다.

## 영향
- 리뷰 요청 방식은 GitHub 의 기본 Codex OAuth 연동 동작에 맞춰 단순해진다.
- 저장소 안에서 별도 prompt 세트나 API key secret 을 운영하지 않는다.
- 현재는 `Verify` 성공 후에도 기본 자동 리뷰 요청 코멘트를 남기지 않는다.
- 자동 리뷰 요청을 복구하면 `Verify` 성공이 기본 리뷰 요청의 선행 조건이 된다.
- 복구 후 PR 대화에는 기본 자동 요청이 1개 코멘트로만 남아, 리뷰 시작점이 더 읽기 쉬워진다.
- 기본 번들은 1개 코멘트로 고정하되, 필요하면 추가 관점을 같은 방식으로 확장할 수 있다.
- `COMMENTER_PAT` 는 저장소 secret 으로만 보관하고, 범위는 이 저장소의 `Issues: write`, `Pull requests: write` 수준으로 최소화한다.
- workflow 댓글 본문과 문서 예시는 같은 프롬프트 구조를 공유해야 한다.

## 기본 프롬프트 형식
- 기본 자동 코멘트는 이 형식을 따른다.
- 리뷰 범위: 이 PR의 최신 변경과 판단에 필요한 최소 주변 문맥만 본다.
- 5개 관점 섹션: `architecture`, `security`, `docs`, `feature`, `qa` 순서로 답하게 한다.
- 낮은 우선순위: 순수 스타일 취향 차이, 관점 밖의 취향성 구조 선호, 기능과 직접 연결되지 않는 미세 성능 조정을 공통 낮은 우선순위 항목으로 둔다.
- 출력 형식: findings 먼저, 심각도 높은 순서, 관점 범위를 벗어난 지적은 치명적 연관성이 있을 때만 포함, 근거가 있으면 파일 경로와 라인 번호, 각 섹션에서 문제가 없으면 `문제 없음` 또는 residual risk만 짧게, 한국어 작성.

## 관점별 상세 프롬프트
### architecture
- 우선해서 볼 항목:
  - 구조 경계와 책임 분리가 흐려지지 않는지 본다.
  - 불필요한 결합, 규칙 분산, 우회 경로가 생기지 않는지 본다.
  - 단기 동작은 맞아도 장기 유지보수 비용이 급격히 커지는 변경을 찾는다.
- 낮은 우선순위:
  - 순수 스타일 취향 차이
  - 기능 동작과 직접 연결되지 않는 미세 성능 조정
  - 구조 문제와 무관한 테스트 취향

### security
- 우선해서 볼 항목:
  - 권한, 토큰, 시크릿, 신뢰 경계가 과도하거나 우회 가능하지 않은지 본다.
  - 외부 입력이나 PR 변경이 자동화 경로를 악용할 수 없는지 본다.
  - 권한 설정이 최소 권한 원칙을 벗어나지 않는지 본다.
- 낮은 우선순위:
  - 구조 취향 차이
  - 문서 표현 품질 자체
  - 보안 영향이 없는 일반 성능 문제

### docs
- 우선해서 볼 항목:
  - 문서와 실제 동작이 어긋나지 않는지 본다.
  - `AGENTS.md`, `README`, `index`, design doc 사이의 라우팅과 설명이 일관적인지 본다.
  - 새 규칙이나 운영 변경에 필요한 설계 기록과 실행 기록이 빠지지 않았는지 본다.
- 낮은 우선순위:
  - 의미를 바꾸지 않는 단순 오탈자
  - 문서와 무관한 코드 구조 취향
  - 문서 영향이 없는 미세 구현 이슈

### feature
- 우선해서 볼 항목:
  - 사용자 흐름과 핵심 목적이 실제로 충족되는지 본다.
  - 저장, 조회, 상태 전이가 기능 의도와 어긋나지 않는지 본다.
  - 기능은 동작해도 회귀 위험이나 핵심 분기 누락이 남아 있지 않은지 본다.
- 낮은 우선순위:
  - 순수 구조 취향 차이
  - 기능과 직접 연결되지 않는 미세 성능 조정
  - 보안 전용 세부 기준

### qa
- 우선해서 볼 항목:
  - 핵심 사용자 시나리오를 막는 검증 공백이 없는지 본다.
  - `loading`, `empty`, `error`, `disabled` 같은 상태가 빠지지 않았는지 본다.
  - `verify`, 테스트, 빌드가 실제 회귀를 잡아낼 만큼 연결되어 있는지 본다.
- 낮은 우선순위:
  - 구현 스타일 차이
  - 검증과 직접 연결되지 않는 구조 선호
  - QA 영향이 없는 순수 문구 수정

## 종합 코멘트 예시 구조
```text
@codex 이 PR의 최신 변경을 종합 리뷰해주세요.

이번 리뷰 범위는 이 PR의 최신 변경과 판단에 필요한 최소 주변 문맥으로 제한하세요.

다음 5개 섹션으로 나눠서 답하세요.
1. architecture
2. security
3. docs
4. feature
5. qa

출력 형식:
- findings 를 먼저, 심각도 높은 순서로 정리하세요.
- 각 섹션마다 명시적 문제가 없으면 문제 없음 또는 남은 리스크만 짧게 적으세요.
- 관점 범위를 벗어난 지적은 치명적 연관성이 있을 때만 포함하세요.
- 근거가 있으면 파일 경로와 라인 번호를 적으세요.
- 스타일 취향보다 버그, 회귀, 위험한 가정, 누락된 검증을 우선하세요.
- 리뷰 본문은 한국어로 작성하세요.
```

## 운영 메모
- `request_codex_review` job 은 현재 비활성화되어 있다. 복구하려면 `.github/workflows/verify.yml` 에서 `if: ${{ false }}` 를 `if: github.event_name == 'pull_request'` 로 되돌린다.
- `COMMENTER_PAT` 가 비어 있으면 자동 댓글 job 은 notice 만 남기고 건너뛴다.
- fine-grained PAT 를 권장하고, 발급 계정은 Codex 와 연결된 같은 GitHub 계정을 사용한다.
