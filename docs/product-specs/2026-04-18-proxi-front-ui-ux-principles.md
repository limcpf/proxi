# proxi 프런트 UI UX 원칙

## 목표
- 화면 목적, 액션 위계, 정보 공개 순서를 일정하게 유지해 사용자 피로를 줄인다.

## 기본 방향
- one page, one purpose
- one primary action
- progressive disclosure
- summary first, detail later

## 화면 목적
- 한 페이지는 하나의 중심 목적을 가져야 한다.
- 페이지 상단만 읽어도 이 화면이 무엇을 위한 곳인지 알 수 있어야 한다.
- 화면 안에 여러 목적이 섞이면 route 나 section 책임을 다시 나눈다.

## 액션 위계
- primary action 은 하나를 기본값으로 둔다.
- secondary action 은 시각적 대비를 낮춘다.
- 파괴적 액션은 멀리 두고 의도를 다시 확인하게 만든다.
- 동등한 중요도의 버튼을 여러 개 나열하지 않는다.

## 점진적 공개
- 자주 쓰는 정보와 행동을 먼저 보여준다.
- 상세 정보, 고급 옵션, 예외 설정은 접힌 영역이나 보조 진입점으로 뒤로 뺀다.
- 사용자가 판단에 필요한 최소 정보는 접지 않는다.

## 정보 밀도
- 첫 화면에서는 요약, 상태, 다음 행동을 우선한다.
- 목록은 스캔 가능한 밀도를 유지하고 장식보다 비교 가능성을 우선한다.
- 긴 설명은 짧은 제목과 보조 텍스트로 쪼갠다.
- 한국어 문장은 짧게 끊고, 한 줄 폭을 지나치게 넓히지 않는다.

## 상태 표현
- loading, empty, error, disabled 상태를 기본 설계 범위에 포함한다.
- empty 상태는 아무것도 없다는 사실보다 다음 행동을 먼저 설명한다.
- error 상태는 문제와 다음 조치를 같이 보여준다.
- disabled 상태는 왜 막혔는지 근거를 같이 제공한다.

## 폼과 입력
- 폼은 입력 순서가 자연스럽게 읽히도록 배치한다.
- 검증 메시지는 필드 가까이에 둔다.
- submit 은 페이지 목적과 연결된 하나의 문장으로 표현한다.
- 선택지가 많을수록 기본값과 추천 경로를 먼저 보여준다.

## 수용 기준 체크리스트
- 페이지 목적이 한 문장으로 설명된다.
- primary action 이 하나로 보인다.
- loading / empty / error / disabled 상태가 빠지지 않았다.
- 요약 정보와 상세 정보의 공개 순서가 분리되어 있다.
- 같은 중요도의 액션이 과하게 병렬 배치되지 않았다.
