# 글 도메인 요구사항 수립 체크리스트

## 문서 목적
- `글` 도메인을 구현하기 전에 사람이 먼저 결정해야 할 제품 요구사항을 정리한다.
- 이 문서는 최종 사양서가 아니라, 구현 가능한 사양서를 만들기 위한 질문 목록이다.
- 답이 정해진 항목은 이후 별도 제품 사양 문서나 실행 계획에 구체적인 수용 기준으로 옮긴다.

## 1. 도메인 목표
- 사용자가 `글`을 작성하는 주된 목적은 무엇인가?
- `글`은 누구에게 보여주는가? 본인만 보는 기록인가, 팀 공유 문서인가, 공개 콘텐츠인가?
- 첫 버전에서 반드시 가능해야 하는 핵심 행동은 무엇인가?
- 첫 버전에서 명시적으로 하지 않을 것은 무엇인가?
- 성공 여부는 어떤 사용자 행동으로 판단하는가?

## 2. 사용자와 권한
- 글을 만들 수 있는 사용자는 누구인가?
- 글을 볼 수 있는 사용자는 누구인가?
- 글을 수정할 수 있는 사용자는 작성자뿐인가, 다른 역할도 가능한가?
- 글을 삭제할 수 있는 사용자는 누구인가?
- 로그인하지 않은 사용자가 글을 볼 수 있는 경우가 있는가?
- 작성자 탈퇴, 권한 변경, 소유권 이전 상황은 어떻게 처리하는가?

## 3. 글의 생명주기
- 글은 어떤 상태를 가지는가? 예: 초안, 게시됨, 숨김, 삭제됨
- 작성 중 자동 저장이 필요한가?
- 게시 전 미리보기나 검토 단계가 필요한가?
- 수정 후 즉시 반영되는가, 다시 게시해야 하는가?
- 삭제는 즉시 영구 삭제인가, 복구 가능한 소프트 삭제인가?
- 삭제된 글의 URL이나 목록 노출은 어떻게 처리하는가?

## 4. 작성과 수정
- 글 작성에 필요한 최소 필드는 무엇인가? 예: 제목, 본문, 요약, 태그
- 제목 없이 저장할 수 있는가?
- 본문 없이 초안 저장이나 게시가 가능한가?
- 본문은 일반 텍스트, Markdown, rich text 중 무엇인가?
- 이미지, 파일, 링크, 임베드 같은 부가 콘텐츠를 지원하는가?
- 수정 이력이나 이전 버전 복구가 필요한가?
- 동시에 여러 사용자가 같은 글을 수정하는 상황을 지원해야 하는가?

## 5. 목록, 조회, 탐색
- 사용자는 어떤 기준으로 글 목록을 찾는가? 예: 최신순, 작성자, 상태, 태그
- 검색이 첫 버전에 필요한가?
- 목록에서 보여야 하는 요약 정보는 무엇인가?
- 글 상세 화면에서 가장 중요한 primary action 은 무엇인가?
- 빈 목록 상태에서는 어떤 다음 행동을 안내해야 하는가?
- 존재하지 않거나 볼 수 없는 글에 접근하면 어떤 메시지를 보여줘야 하는가?

## 6. 삭제와 위험 동작
- 삭제 전 확인이 필요한가?
- 삭제 확인 문구는 어떤 위험을 명확히 알려야 하는가?
- 삭제 후 되돌리기나 복구 경로가 필요한가?
- 이미 공유된 링크, 댓글, 첨부파일, 검색 결과는 삭제 후 어떻게 되는가?
- 글 삭제가 다른 데이터에 미치는 영향은 무엇인가?

## 7. 데이터 규칙
- 제목, 본문, 요약, 태그의 길이 제한은 어떻게 둘 것인가?
- 허용하지 않을 콘텐츠나 입력 패턴이 있는가?
- 시간 정보는 무엇을 기록해야 하는가? 예: 생성일, 수정일, 게시일, 삭제일
- 글 ID 는 사용자에게 노출되는가?
- URL slug 가 필요한가? 필요하다면 중복과 수정은 어떻게 처리하는가?
- 정렬과 필터 기준에 필요한 필드는 무엇인가?

## 8. 오류와 예외
- 저장 실패 시 사용자가 작성 중인 내용을 잃지 않게 할 방법은 무엇인가?
- 권한이 없는 수정, 삭제, 조회 요청은 어떻게 보여주는가?
- 네트워크 오류, 서버 오류, 검증 오류를 화면에서 어떻게 구분하는가?
- 너무 긴 본문이나 금지된 입력은 어느 시점에 막는가?
- 중복 제출이나 빠른 연속 클릭은 어떻게 처리하는가?

## 9. 구현 전 수용 기준 초안
- 사용자는 필수 입력을 채워 새 글을 만들 수 있다.
- 사용자는 본인이 접근 가능한 글 목록을 볼 수 있다.
- 사용자는 글 상세 내용을 볼 수 있다.
- 사용자는 권한이 있는 글을 수정할 수 있다.
- 사용자는 권한이 있는 글을 삭제할 수 있다.
- loading, empty, error, disabled 상태가 각 주요 화면에 존재한다.
- 권한이 없는 행동은 서버와 화면 양쪽에서 막힌다.
- 저장 실패 시 작성 중인 입력이 사라지지 않는다.

## 10. 구현 전에 확정할 답
- 첫 버전의 글 상태 모델은 무엇인가?
- 삭제는 hard delete 인가 soft delete 인가?
- 본문 포맷은 무엇인가?
- 인증과 작성자 권한은 어디까지 전제할 것인가?
- 목록과 상세 화면의 primary action 은 각각 무엇인가?
- `shared` 에 둘 계약과 `back` 내부 모델의 경계는 어디인가?
- 첫 vertical slice 의 범위는 어디까지인가?

## 11. Shared 에 답해야 할 질문
- `@proxi/shared` 에 노출할 글 도메인 public API 는 무엇인가?
- `shared` 에 둘 것은 API 계약, DTO, enum, zod schema 중 어디까지인가?
- `back` 내부 DB 모델과 `front` 화면 모델을 분리하기 위해 어떤 타입 이름을 사용할 것인가?
- 글 ID, 작성자 ID, slug 같은 식별자는 branded type 으로 둘 것인가?
- 글 상태는 어떤 enum 또는 union 으로 정의할 것인가?
- 작성 요청, 수정 요청, 목록 응답, 상세 응답, 삭제 응답의 최소 shape 는 무엇인가?
- 검증 schema 의 에러 메시지는 사용자 노출용인가, 개발자/서버용인가?
- 날짜와 시간은 string ISO 형식으로 공유할 것인가, 각 앱 내부에서 Date 로 변환할 것인가?
- `shared` 에 포함하면 안 되는 front 전용 UI 상태나 back 전용 DB 필드는 무엇인가?
- shared 계약 변경 시 어떤 테스트가 깨져야 하는가?

## 12. Back 에 답해야 할 질문
- 글 도메인 Nest module 이름과 경계는 무엇인가? 예: `WritingModule`, `PostsModule`, `ArticlesModule`
- 첫 API endpoint 는 무엇부터 열 것인가? 예: create, list, detail, update, delete
- REST path 는 어떤 형태인가? 예: `/writing/posts`, `/posts`
- 인증이 아직 없을 때 작성자와 권한은 어떻게 임시 모델링할 것인가?
- DB/Prisma 를 이번 vertical slice 에 포함할 것인가, 아니면 in-memory 저장소로 시작할 것인가?
- soft delete 를 쓰면 DB 필드와 조회 필터는 어떻게 둘 것인가?
- validation 은 controller DTO, shared zod schema, pipe 중 어디에서 수행할 것인가?
- 권한 없음, 없음, 검증 실패, 충돌 상황의 HTTP status 는 무엇인가?
- service, repository, controller 책임은 어떻게 나눌 것인가?
- 목록 조회의 pagination, sort, filter 기본값은 무엇인가?
- 수정 동시성 제어가 필요한가? 필요하다면 version 또는 updatedAt 조건을 사용할 것인가?
- 백엔드 테스트는 unit, controller, integration 중 어디까지 첫 버전에 포함할 것인가?
- 로그에 남겨야 할 글 도메인 이벤트는 무엇인가? 예: created, updated, deleted

## 13. Front 에 답해야 할 질문
- 첫 화면은 목록 중심인가, 작성 폼 중심인가, 상세 중심인가?
- route 구조는 어떻게 둘 것인가? 예: `/writing`, `/writing/new`, `/writing/$postId`
- 글 목록의 primary action 은 무엇인가?
- 글 상세의 primary action 은 무엇인가?
- 작성/수정 폼을 같은 컴포넌트로 공유할 것인가, 분리할 것인가?
- 본문 입력 UI 는 textarea, Markdown editor, rich text editor 중 무엇인가?
- URL 상태로 올릴 값은 무엇인가? 예: page, status, keyword, tag, sort
- TanStack Query key 는 어떤 규칙으로 만들 것인가?
- mutation 성공 후 어떤 query 를 invalidate 할 것인가?
- 작성 중 이탈, 저장 실패, 중복 제출을 어떻게 막거나 안내할 것인가?
- 삭제 confirmation 은 modal, inline confirm, separate route 중 무엇으로 할 것인가?
- loading, empty, error, disabled 상태의 문구는 무엇인가?
- optimistic update 를 쓸 것인가, 서버 응답 후 갱신만 할 것인가?
- `@proxi/shared` 타입을 화면 상태에 직접 쓸 것인가, front 전용 view model 로 변환할 것인가?
- 첫 프런트 테스트는 어떤 사용자 흐름을 반드시 포함할 것인가?

## 14. 프로젝트별 최소 산출물
- Shared: 글 ID, 글 상태, create/update 요청 schema, list/detail 응답 타입, 계약 테스트
- Back: 글 module, controller, service, 저장소 경계, create/list/detail/update/delete 테스트
- Front: 글 route, 목록 화면, 작성/수정 폼, 삭제 확인 흐름, query/mutation 테스트
- Docs: 확정된 제품 사양, API 계약 근거, 첫 vertical slice 실행 계획
