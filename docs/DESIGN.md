# 설계 문서 운영 기준

## 이 문서를 읽어야 하는 경우
- 모듈 경계가 바뀐다.
- DB 스키마나 데이터 흐름이 바뀐다.
- 외부 시스템 연동이 추가되거나 변경된다.
- 신뢰성 또는 보안 가드레일이 바뀐다.
- 여러 파일에 걸친 구조적 변경이라 작업 중 판단 기준이 필요하다.

## 설계 문서가 필요한 작업
- 되돌리기 어려운 구조 변경
- 여러 대안을 비교해야 하는 변경
- 운영 비용이나 장애 양상에 영향을 주는 변경
- 규칙을 팀 공통 기준으로 승격하려는 변경

## 권장 템플릿
- `상태`: `proposed`, `accepted`, `deprecated`
- `배경`: 현재 제약과 문제
- `결정`: 채택한 방식
- `대안`: 버린 선택지와 이유
- `영향`: 코드, 운영, 테스트, 문서에 미치는 영향
- `후속 작업`: 실행 계획 또는 기술 부채 링크

## 작성 규칙
- 문서 하나에는 한 가지 결정만 담는다.
- 구현 세부사항보다 경계와 이유를 우선한다.
- 관련 경로와 선행 문서를 명시한다.
